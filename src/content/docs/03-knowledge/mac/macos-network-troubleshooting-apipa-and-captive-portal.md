## 1. 公共 WiFi 不弹出认证页面 (Captive Portal 失败)

连接酒店或机场的公共 WiFi 时，通常需要弹出一个 Web 页面进行短信或账号认证。如果不弹出，设备将一直处于无网状态。

### 1.1. 诊断测试地址

在浏览器中手动访问 Apple 的官方探测地址，看是否被劫持重定向至认证页：
👉 `https://captive.apple.com/`

### 1.2. macOS 深度修复步骤

1. **清理冗余 DNS (关键)**
   - 点击状态栏 Wi-Fi 图标 -> `Wi-Fi 设置` -> 找到当前连接的 WiFi 点击 `详细信息`。
   - 切换到 **DNS** 选项卡。删除右侧列表中的所有自定义 DNS IP（如 8.8.8.8 或 114.114.114.114）。删到最后，系统会自动恢复为当前路由器分配的网关 IP。
2. **关闭全局代理**
   - 同一设置页面下，切换到 **代理** 选项卡，确保关闭所有代理开关（防认证流量被代理软件拦截）。
3. **放行 Safari 弹出窗口**
   - 打开 Safari 浏览器 -> `设置 (Preferences)` -> `网站`。
   - 左侧选择 `弹出式窗口`，将右侧“访问其他网站时”的选项改为 **允许**。

*完成上述操作后，断开并重新连接 WiFi，或者在 Safari 中新建 Tab 页，即可触发认证窗口。*

## 2. IP 地址变为 169.254.*.* (APIPA) 现象深度解析

当你看到以 **169.254** 开头的 IP 地址时，这并不是中毒，而是表示：**你的设备尝试联网，但没能从路由器（DHCP 服务器）获取到有效的 IP 地址**。

这种地址背后的技术统称为 **APIPA (Automatic Private IP Addressing，自动私有 IP 寻址)** 或 **链路本地地址 (Link-Local Address)**。

### 2.1. 1 触发机制

在正常网络中，设备接入后会发出 DHCP 广播请求分配 IP。如果：

- 路由器死机、DHCP 服务关闭。
- 网线物理损坏或 Wi-Fi 密码/验证机制有隐性错误。
操作系统为了不让网卡完全罢工，会 **强行自定一个 169.254.x.x 的地址** 给自己。

### 2.2. 2 它的用途与限制

- **用途**：当两台电脑直接用一根网线互联且都不设静态 IP 时，它们都会拿到 169.254 的地址，此时它们之间可以正常进行局域网文件传输。
- **限制**：该地址 **绝对无法跨越路由器路由**，即意味着 **彻底无法访问互联网**。

### 2.3. 3 恢复策略矩阵

| 故障现象 | 潜在原因 | 修复动作 |
| :--- | :--- | :--- |
| **物理连接异常** | 网线松动或 Wi-Fi 信号极度衰减导致丢包。 | 重新插拔网线，或让设备靠近路由器源头。 |
| **DHCP 服务假死** | 路由器长期运行导致分配模块卡死。 | **最有效：直接重启路由器硬件**。 |
| **系统 IP 租期滞后** | 电脑卡在旧的 DHCP 请求状态未释放。 | 在终端中尝试重置网络接口 (如 Windows 执行 `ipconfig /release & renew`)。 |
| **安全策略拦截** | 强效防火墙或企业级杀软拦截了 UDP 67/68 端口流量。 | 暂时关闭防火墙或安全卫士进行排查测试。 |
